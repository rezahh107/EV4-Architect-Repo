## Purpose

`/build-tree` converts the audited `/recommend` result into an Elementor Structure Panel plan.

It is not an implementation stage. It creates the structural blueprint that a builder can follow in Elementor V4.

## Core Rule

Build from the audited recommendation, not from visual preference.

Stage 7 may organize containers, widgets, names, classes, editable content, decoration layers, overlay stages, and responsive structural intent.

Stage 7 must not invent a new architecture, re-score candidates, override Stage 5, override Stage 6, write final CSS, write implementation code, or silently drop unknowns.

## Naming Convention

Stage 7 uses the EV4 BEM-like naming convention:

```text
[section-role]__[content-group]--[variant]
```

Examples:

```text
hero__copy--primary
hero__visual--dashboard
feature-grid__card--default
feature-grid__card--highlighted
pricing__card--featured
smart-home__connector-layer--desktop
```

Rules:

- use lowercase Latin letters, numbers, and hyphens;
- use `__` between section/block and content group;
- use `--` only for a variant or state;
- do not encode pixel values, colors, coordinates, or temporary implementation details into class names;
- do not create global classes for one-off coordinates;
- do not use vague names such as `box-1`, `left-side`, `nice-card`, `main-wrapper`, or `visual-div`;
- meaningful repeated items should share a base class and use variants only when role or state differs;
- Structure Panel labels may be human-readable, but class names must remain machine-readable.

## Structure Panel Label Convention

Each tree node must include both:

```text
structure_label: human-readable label for Elementor Structure Panel
class_name: machine-readable EV4 class name if class is needed
```

Example:

```text
structure_label: Hero / Copy / Primary
class_name: hero__copy--primary
```

Labels must help a human builder navigate the Structure Panel. Class names must help design-system reuse and audit.

## Tree Depth and Wrapper Budget

Stage 7 must minimize unnecessary wrappers.

Default limits:

- target maximum depth: 4 structural levels below the section root;
- hard warning at depth 5;
- blocker if depth exceeds 5 without justification;
- avoid one-child containers unless they have a named structural purpose;
- every extra wrapper must have a justification.

Allowed wrapper justifications:

```text
normal-flow grouping
responsive order control
overlay containment
decoration isolation
repeated component boundary
design-system reuse
widget-native requirement
accessibility/semantic grouping
```

Forbidden wrapper justifications:

```text
looks cleaner
maybe useful later
because Elementor needs it
for styling only, with no reusable class or scoped need
unknown reason
```

## Wrapper Justification Table

If the tree contains any container with a single child, any visual-only wrapper, or any overlay stage, Stage 7 must include:

```text
WRAPPER JUSTIFICATION TABLE

- structure_label
- reason
- evidence/source
- can_remove_if_implementation_simplifies: yes/no
```

## Widget Constraint Rules

Stage 7 must preserve editability.

Meaningful content should remain in editable Elementor-native widgets when practical:

- heading text → Heading widget or equivalent editable heading;
- paragraph/body copy → Text Editor widget or equivalent editable text;
- CTA → Button widget or link-capable native widget;
- image with meaning → Image widget with alt-text requirement carried forward;
- repeated card → container/component/loop candidate according to Stage 6 authorization;
- icon with label → Icon or Icon Box only if it preserves content structure;
- FAQ/carousel/counter/filter → widget-native candidate only if Stage 6 allowed the interaction family.

HTML/SVG widgets are allowed only when:

- Stage 6 permits them;
- content is decoration-only or a controlled visual layer;
- meaningful text is not trapped inside static markup;
- responsive and accessibility unknowns are carried forward.

## Overlay and Decoration Rules

Overlays must be contained.

If any overlay, connector, floating card, SVG line, absolute decoration, z-index layer, or decorative visual stage is used, the tree must include a named relative stage:

```text
[section-role]__stage--relative
```

Overlay children must live inside this stage, not at page root.

Meaningful content must not depend on an overlay layer for reading order.

Decoration layers must be marked as one of:

```text
decoration-only
visual-core
unknown-meaningfulness
```

If meaning is unknown, Stage 7 must carry the unknown forward to `/implementation`.

## Responsive Structure Contract

Stage 7 describes structural responsive intent, not final responsive CSS.

For each major group, include:

```text
desktop_structure:
tablet_structure:
mobile_structure:
responsive_order:
hide_or_simplify_allowed:
responsive_unknowns:
```

Rules:

- do not invent exact breakpoints unless provided;
- do not hide meaningful content on mobile unless Stage 6 explicitly allowed it;
- decorative connector layers may be simplified or hidden only if marked decoration-only;
- if mobile behavior is unknown, carry it forward rather than resolving it silently;
- reading order must remain logical when visual order changes.
